# OCR MCP Server Architecture

## Goal

Build a high-performance Go-based MCP server that enables **text-only
LLMs** (DeepSeek, Kimi, GLM, Minimax, etc.) to understand images by
converting them into structured text before the LLM receives the
context.

The LLM never sees the image directly---it only receives structured
Markdown/JSON generated by the OCR pipeline.

------------------------------------------------------------------------

# High-Level Architecture

``` text
                OpenCode
                    │
                    ▼
           Go MCP Server
                    │
         ┌──────────┴──────────┐
         ▼                     ▼
   OCR Service           Vision Service (Optional)
 (PaddleOCR)          (Qwen2.5-VL / Florence-2)
         │                     │
         └──────────┬──────────┘
                    ▼
          Structured Markdown/JSON
                    │
                    ▼
              Text-only LLM
```

------------------------------------------------------------------------

# Why Separate Services?

Starting Python and loading OCR models for every request adds **1--3
seconds** of latency.

Instead:

-   Start the OCR service once.
-   Load PaddleOCR into memory once.
-   Keep it running.
-   The Go MCP server communicates with it over HTTP or gRPC.

Benefits:

-   Warm models
-   Lower latency
-   Easier scaling
-   Independent deployments

------------------------------------------------------------------------

# Request Lifecycle

``` text
User uploads image
        │
        ▼
OpenCode
        │
        ▼
Text-only LLM
        │
        ▼
Calls MCP Tool
        │
        ▼
Go MCP Server
        │
        ├── Validate request
        ├── Compute image hash
        ├── Check cache
        ├── Preprocess image
        ├── Send image to OCR Service
        ├── Receive OCR result
        ├── (Optional) Send to Vision Service
        ├── Merge outputs
        ├── Format Markdown/JSON
        └── Return response
        │
        ▼
LLM answers user
```

# OCR Service

Responsibilities:

-   Load PaddleOCR at startup.
-   Accept image requests.
-   Run OCR.
-   Return JSON.

Example:

``` json
{
  "text": "Invoice #123\nAmount ₹500",
  "confidence": 0.98
}
```

------------------------------------------------------------------------

# Vision Service (Optional)

Responsibilities:

-   Detect UI layout.
-   Describe screenshots.
-   Identify buttons, inputs, modals.
-   Produce semantic descriptions.

------------------------------------------------------------------------

# Caching

Compute:

``` text
SHA256(image)
```

If already processed:

-   Skip OCR
-   Return cached result immediately

Supports Redis or in-memory cache.

------------------------------------------------------------------------

# Worker Pool

``` text
Go MCP Server

Request
   │
   ▼
Worker Pool
   ├── Worker 1
   ├── Worker 2
   ├── Worker 3
   └── Worker N
```

Allows multiple images to be processed concurrently.

# OCR Provider Interface

``` go
type OCRProvider interface {
    ExtractText(ctx context.Context, image []byte) (*OCRResult, error)
}
```

-   PaddleOCR
-   Tesseract
-   Surya OCR

-   AWS Textract
